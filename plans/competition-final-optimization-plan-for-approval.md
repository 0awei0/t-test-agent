# AI 测试官比赛最终优化计划（待确认）

## 1. 目标与当前基线

- 比赛方向：方向二「AI 测试官」。
- 截止时间：2026-07-19 23:59:59；内部目标仍为 2026-07-18 前完成提交。
- 当前版本：`main` 已推送至 `origin/main`，最新提交为 `7161599 Add per-MR Agent replay catalog`。
- 当前能力：8 个合成 TAPD 需求与 8 个工蜂 MR 一一对应，每个 MR 均有独立的 `agent-strict` 真实运行脱敏回放。
- 当前验证：后端 97 项单元测试通过；前端已有 typecheck/build，但尚无自动化交互测试。
- 优化目标：不再扩展平台型功能，集中提高比赛交付可信度、评委观看效率、自动验收覆盖和最终提交完整性。

## 2. 执行原则

1. 本计划确认前只保存计划，不修改产品代码。
2. 确认后严格按批次执行；每批只解决一个核心问题。
3. 每批修改后先运行对应验证，再提交并推送 `origin/main`。
4. 不提交 `.env`、访问参数、Webhook、真实 TAPD/MR、原始日志、截图或完整运行产物。
5. 线上继续使用脱敏静态回放；真实执行只在本地隔离工作区进行。
6. 最终稳定部署完成后只申请一次长期链接重新绑定，开发阶段使用临时预览链接。

### 目标

让评委既能看到真实事件逐步发生，也能在有限时间内快速到达关键决策，不必等待固定延迟播放完整个复杂案例。

### 预计修改

- `frontend/src/App.tsx`
- `frontend/src/styles.css`
- 前端 E2E

### 工作项

- [ ] 增加暂停/继续、重新播放、1×/2×速度和“跳到结论”。
- [ ] 展示当前事件序号、总事件数和播放状态。
- [ ] 提供“一键评委演示”：默认进入 `task-45`，以 2×播放关键过程。
- [ ] 保留 `prefers-reduced-motion` 支持。

## 7. 批次四：强化 Agent 真实性与安全证据

### 目标

减少“预制动画”的观感，并通过真实事件证明 Agent 会自我修正、安全策略会实际阻断危险动作。

### 工作项 A：运行来源卡片

- [ ] 展示运行模式、run ID、执行时间和是否通过严格工具检查。
- [ ] 展示实际工具调用数、策略事件数、命令数、生成测试数和证据数。
- [ ] 明确区分 Agent 自主工具调用与系统确定性安全兜底。

### 工作项 B：自我修正链路

- [ ] 将“测试失败 → 读取日志 → 修改测试/策略 → 重跑 → 新结论”标记为一次自我修正 episode。
- [ ] 保留被新结果替代的失败轨迹，但最终统计只使用最新有效结果。
- [ ] 在复杂案例中提供可直接定位的自我修正证据。

### 工作项 C：实际安全拦截

- [ ] 在合成场景中安全模拟一次越权路径、非白名单命令或远端写请求。
- [ ] 由现有安全工具真实拒绝该动作，并生成 `safety_check/blocked` 事件。
- [ ] 前端展示请求、拒绝原因和安全替代方案。
- [ ] 不实际执行危险命令，不访问真实业务资源。

### 工作项 D：准确描述上下文能力

- [ ] 将当前“上下文记忆压缩”改为“运行上下文摘要 / 归档压缩”。
- [ ] 明确当前摘要在运行后形成，原始 diff、日志和证据可按需回读。
- [ ] 本轮不宣称已实现运行中 context-window 压缩恢复。

### 验收标准

- 默认复杂案例能看到运行来源和完整自我修正链。
- 安全案例中存在由真实守卫产生的 blocked 事件。
- 页面和提交文案不存在对“实时执行”或“运行中记忆压缩”的夸大表述。
- Python 测试、前端 E2E 和比赛门禁通过。

### 预期提交

```text
Strengthen Agent provenance and safety evidence
```

## 10. 明确暂缓项

以下能力本次比赛提交前不进入主线：

- 真实 TAPD、工蜂的在线强依赖接入。
- 云端任意仓库和任意命令执行。
- 自动评论、提交、推送、合并或发布真实业务仓库。
- 完整多租户测试工作台。
- 新增更多普通 MR 案例。
- 真正的运行中 context-window 压缩与恢复。
- 大规模视觉重构。

如主线提前完成，可补充一份脱敏的真实适配器证明，但不得让现场演示依赖内网和账号状态。

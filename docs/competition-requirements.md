## 3. 方向一：AI 业务洞察

### 3.1 方向定位

目标是让 AI 同时理解文本知识和结构化数据，并将融合后的洞察用更直观、可交互、有感染力的方式呈现出来。

### 3.2 背景与问题

企业运营中的知识资产主要包括两类：

- 文本型资产：产品文档、SOP、会议纪要、研报、PRD、培训材料等。
- 数据型资产：OKR 指标、用户行为数据、经营指标、运营看板等。

当前痛点包括：

- 跨域关联困难：数据趋势难以对应文档中的策略或决策背景。
- 洞察碎片化：只能看到“是什么”，难以解释“为什么”。
- 知识传递低效：新人难以从分散资产中建立整体业务认知。

### 3.3 赛题要求

作品应重点解决两个层面的问题：

- 知识融合层：AI 不只是分别查询文档和数据，而是能够让文本知识与结构化数据相互印证、交叉推理，形成“数据有语境、文档有数据支撑”的融合洞察。
- 洞察呈现层：融合后的洞察不能只停留在传统 PPT 或报表中，鼓励探索可对话、可追问、可交互的新型叙事载体。

### 3.4 可选思路

- 先选择一个真实业务场景，例如产品迭代复盘、客户流失分析、季度经营分析、竞品情报整合等。
- 准备该场景下的典型文本样本与数据样本。
- 设计 AI 如何融合数据和文档两类业务知识。
- 设计洞察呈现载体，例如交互式 Dashboard、动态 Storytelling 页面、知识图谱可视化、智能简报等。

### 3.5 工具与平台

工具和平台不做限制。参赛者可以自由选择 AI 基础设施、开发框架和数据源。官方鼓励基于真实工作流（个人、团队或业务）的可演示原型。

推荐使用狍子（Paozi）、Vedas Agent 平台能力，包括：

- 知识库与记忆系统。
- Agent infra：MCP、Skill、记忆、RAG 等工具。
- 自动化任务与订阅监控。
- 用 Agent 制作演示内容。

### 3.6 作品提交标准

- 作品形态不限，可为交互式 Dashboard、动态 Storytelling 页面、知识图谱可视化、智能简报等。
- 鼓励图形化界面或实时展示方式，用于呈现过程与结果。
- Demo 必须可实际运行，评审环节需要现场演示。
- 作品标题命名格式：`方向一【AI业务洞察】XXXXXXXXXX`

## 4. 方向二：AI 测试官

### 4.1 方向定位

目标是构建一个覆盖后端逻辑到前端体验的全链路自动化测试 Agent。它需要理解代码变更意图，自主规划测试策略，完成端到端验证，并输出人可以直接决策的测试报告。

### 4.2 背景与核心问题

测试是软件交付中耗费人力较多、同时也最适合提效的环节。赛题关注的问题是：能否让 Agent 不依赖详尽人工测试清单，只通过简单测试指令就能读懂代码、判断影响范围、决定测试策略，并最终完成验证与报告。

### 4.3 赛题要求

作品不能只是“跑脚本、输出日志”，而应形成完整闭环：

- 理解变更：读取代码、需求、PR diff、Bug 信息或线上状态，判断变更意图。
- 规划策略：基于影响范围自主拆解测试场景、选择测试手段和优先级。
- 执行验证：覆盖后端逻辑、接口、数据、前端交互或线上巡检等环节。
- 汇总报告：用简明易懂的方式呈现结论、异常点、可能原因和建议动作。

关键判断标准不是“自动执行了多少脚本”，而是 Agent 是否能在测试清单不完整的情况下自行做判断：

- 输入是否足够简单：用户只给“测一下这个改动”“检查这个需求”“巡检核心链路”等高层指令。
- 理解是否有证据：报告能解释它读到了哪些变更、需求或异常信号，以及为什么认为这些地方有风险。
- 策略是否有取舍：能说明要测什么、不测什么、优先级如何，以及选择单测、接口测试、浏览器测试或巡检的理由。
- 执行是否真实：评审现场可以看到命令、浏览器、日志、截图、失败用例或其他可复核证据。
- 报告是否可决策：开发或值班同学能直接判断通过、阻断、需要补测还是需要修复。

### 4.4 官方启发场景

- 场景 A：开发刚提交代码后，AI 测试官读取改动内容，判断潜在风险点，针对性执行测试并直接反馈。
- 场景 B：产品完成需求文档后，AI 测试官读取需求，拆解测试场景，再检查代码中功能点是否被实现。
- 场景 C：AI 测试官定时巡检线上核心功能，发现异常后整理异常点和可能原因，并推送给开发或值班同学。

三个场景可以拆成不同成熟度：

| 场景 | 输入 | Agent 核心动作 | 输出 | 当前项目优先级 |
| --- | --- | --- | --- | --- |
| A：提交后反馈 | 本地 diff、PR/MR diff、分支范围 | 理解改动意图，定位风险文件和风险分支，选择并执行相关测试 | 面向开发的变更风险和验证报告 | P0，当前 MVP 主线 |
| B：需求到实现核验 | PRD、TAPD story/bug/task、代码仓库 | 拆需求点，映射代码实现，检查功能是否覆盖，生成测试场景 | 需求覆盖矩阵、缺口、建议测试 | P1，接入 TAPD 后做 |
| C：线上定时巡检 | 定时任务、环境信息、核心链路清单 | 周期性跑接口或浏览器链路，异常时归因并通知 | 巡检报告、异常摘要、值班通知 | P2，需调度与稳定环境 |
| 前端体验增强 | 页面 URL、用户路径、账号或测试环境 | 用 Playwright MCP 或本地 Playwright 测试驱动真实浏览器，采集截图/失败证据 | 带截图和步骤的 E2E 报告 | P0，A-fullstack Demo 已覆盖 |

### 4.5 工具与平台

工具和基础设施不做限制。官方推荐使用 Box 平台能力，包括：

- MCP 连接器：例如接入 TGit 读取 PR diff、接入 TAPD 关联 Bug 单等。
- Playwright MCP：需要真实浏览器交互验证时，用于驱动浏览器。
- 定时任务：用于持续巡检场景。
- bot 机器人：用于将测试报告实时推送给开发或值班人员。

### 4.6 作品提交标准

- 作品形态不限，官方推荐 Agent、Skill、插件等形式。
- 鼓励用图形化界面或实时展示方式呈现测试过程与结果。
- Demo 必须可实际运行，评审环节需要现场演示。
- 作品标题命名格式：`方向二【AI测试官】XXXXXXXXXX`

提交材料建议：

- 标题：严格使用 `方向二【AI测试官】XXXXXXXXXX`，标题主体尽量说明“变更理解”“全链路验证”或“自动报告”能力。
- 作品链接：放一个稳定入口，包含 Demo 命令、演示视频或可交互页面。
- 作品简介：200 字以内突出“简单指令输入、自动理解变更、自动规划并执行验证、报告可决策”。
- 现场演示：准备可重复运行的合成 demo 仓库，再准备一个授权的内部代码演示路径作为增强。

现场 Demo 的最低闭环：

1. 展示一个刚提交的改动或 PR diff。
2. 运行一条简单命令，而不是手工给测试清单。
3. Agent 输出风险判断和测试计划。
4. Agent 实际执行测试或浏览器验证。
5. 报告指出通过/失败、失败原因、证据和建议动作。
6. 可选：通过 WeCom bot 推送报告摘要。

当前推荐现场 Demo 路径：

```bash
uv sync --extra codex --extra e2e --group dev
uv run python -m playwright install chromium
uv run ai-test-officer scenario create --scenario A-fullstack --demo-root /tmp/ai-test-officer-scenarios
uv run ai-test-officer scenario run --scenario A-fullstack --demo-root /tmp/ai-test-officer-scenarios --visualize
```

该路径展示本地 diff、后端业务单测、HTTP API、浏览器 checkout 流程、截图证据、Markdown 报告和 HTML 可视化报告。

## 10. 对本项目“测试 Agent”的落地启发

如果本项目选择“方向二：AI 测试官”，建议围绕比赛要求突出以下能力：

- 输入尽量简单：例如只提供 PR、需求文档、分支名、Bug 单或测试目标，Agent 自主完成后续分析。
- 强调变更理解：读取代码 diff、需求说明、相关测试、历史缺陷等，说明为什么这些区域需要测。
- 形成测试策略：明确覆盖范围、测试优先级、测试类型和跳过项原因。
- 能真实执行：至少覆盖一条端到端链路，最好同时展示接口/单测/页面交互/回归巡检中的多种验证能力。
- 报告可决策：输出通过/失败、风险等级、复现步骤、日志摘要、截图/录屏、疑似原因和建议修复方向。
- 可视化呈现：提供测试过程时间线、执行状态、失败定位、截图证据和最终结论，而不是只给命令行日志。
- 演示稳定：评审需要现场运行，Demo 应准备可重复执行的样例仓库、样例需求或样例 PR。
- 合规处理：演示数据和代码样例需要脱敏，避免包含真实敏感业务信息。
